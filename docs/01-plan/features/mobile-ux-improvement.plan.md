## 1. Overview

### 1.1 Purpose

20명 가상 사용자 테스트에서 "모바일에서 좀 불편하다" 피드백이 수집되었다.
ArtRef Finder의 핵심 사용 시나리오인 **러프 스케치 완성 → 실사 레퍼런스 검색 → 참고** 플로우를
모바일 한 손 조작으로도 쉽고 빠르게 완수할 수 있도록 UX를 개선한다.

### 1.2 Background

**현재 앱의 모바일 UX 문제 (코드 분석 기반)**

#### 1.2.1 전체 아키텍처 문제

현재 `/mannequin` 페이지(검색의 핵심)는 모바일에서 `MobileTabView` 컴포넌트로 분기한다.
"포즈 설정" 탭과 "결과" 탭으로 나뉘어 있어 기본 구조는 갖춰졌으나, 각 탭의 콘텐츠가
데스크톱용 레이아웃을 그대로 압축한 형태라 세부 UX가 미흡하다.

#### 1.2.2 상단 헤더 과밀 문제 (고심각)

`mannequin/page.tsx` 헤더(`h-12`)에 아래 요소가 한 줄에 모두 들어있다:
- 로고 (ArtRef)
- ModeTabs (마네킹 / 드로잉)
- TagSearchInput (max-w-sm — 텍스트 검색창)
- 대시보드 링크
- 내 컬렉션 링크
- 로그인/로그아웃 버튼

모바일(360~390px 폭)에서는 이 요소들이 서로 겹치거나 찌그러진다.
`TagSearchInput`은 `flex-1 max-w-sm mx-4`로 배치되어 있어
모바일에서는 너비가 0에 가까워지거나 레이아웃이 깨진다.

#### 1.2.3 "포즈 설정" 탭 UX 문제 (고심각)

모바일 탭의 "포즈 설정" 콘텐츠(`leftPanel`)는 아래 요소를 순서대로 세로로 나열한다:
1. 3D MannequinViewer (Three.js — GPU 집약적, 모바일 성능 저하)
2. 뷰어 하단 도구 바 (반전, 해부학, 체형 토글)
3. 성능 설정 패널 (접기/펼치기)
4. 관절 슬라이더 패널
5. 이미지 업로드 존 ("이미지에서 포즈 추출")
6. 포즈 프리셋 카드 (grid-cols-4)
7. 카메라 앵글 프리셋
8. 자동 태그 패널 3개 (gizmo / cameraAngle / light)
9. 포즈 저장/불러오기 패널
10. 태그 필터 패널 (접기/펼치기, max-h-[40vh] 내부 스크롤)

이 모든 요소가 단일 스크롤 컨테이너에 담겨 있어, 핵심 기능인
**포즈 프리셋 선택** 또는 **태그 필터**에 도달하려면 긴 스크롤이 필요하다.

#### 1.2.4 프리셋 카드 터치 타겟 문제 (중심각)

`PosePresetCards` 의 전신 포즈 그리드는 `grid-cols-4 gap-1.5`이다.
각 카드에는 `p-1.5 rounded-lg` 패딩이 적용되어 있어
실제 터치 타겟이 약 36~38px로, Apple HIG / Material 권장 44px 미충족.

#### 1.2.5 이미지 상세 모달 모바일 레이아웃 문제 (중심각)

`ImageGrid`의 상세 모달은 `flex flex-col md:flex-row` 레이아웃이다.
모바일에서는 상단 이미지 영역 `min-h-[300px]` + 하단 정보 패널이
세로로 쌓이는데, 화면 높이가 짧은 기기에서는 정보 패널이
스크롤 없이는 보이지 않는다.

이미지 하단의 도구 바(반전, 줌, 저장)가 `absolute bottom-3`으로
이미지 위에 오버레이되어 있어, 모바일 하단 Safe Area와 겹칠 수 있다.

#### 1.2.6 태그 필터 접근성 문제 (중심각)

`SearchFilters`의 태그 필터는 접기/펼치기 방식으로 "좌측 패널 스크롤 맨 아래"에 위치한다.
아티스트가 가장 자주 쓰는 검색 방식인 태그 선택이 가장 깊은 곳에 숨어있다.

#### 1.2.7 랜딩 페이지 모바일 CTA 문제 (저심각)

랜딩 히어로의 Primary CTA(`마네킹 움직여보기`)가 3D 마네킹 페이지로 연결된다.
모바일 사용자에게 바로 Three.js 렌더링 페이지를 보내면 초기 로딩이 느리고
GPU 부하로 인해 첫인상이 나빠질 수 있다.

#### 1.2.8 핵심 사용 시나리오 플로우 분석

```
[핵심 사용 시나리오]
러프 스케치 완성
    → 앱 열기 (랜딩 or 직접 /mannequin 진입)
    → 포즈 조건 설정 (프리셋 또는 태그)
    → 결과 탭으로 전환
    → 이미지 탐색
    → 이미지 상세 보기
    → 참고 (다운로드 or 컬렉션 저장)

[현재 마찰 포인트]
1. /mannequin 로딩 시 Three.js 초기화 지연 (모바일 GPU)
2. 포즈 설정 → 프리셋 카드까지 스크롤 필요
3. "포즈 설정" 탭 → "결과" 탭 전환 버튼이 상단에 고정되어 결과 수가 즉시 보임 (이는 장점)
4. 이미지 상세에서 "컬렉션 저장" 버튼이 정보 패널 하단에 위치 (스크롤 필요)
5. 태그 기반 검색은 포즈 설정 패널 가장 아래에 위치 (모바일 대안 진입점 없음)
```

## 2. Scope

### 2.1 In Scope

- [ ] 모바일 헤더 재설계 (요소 과밀 해소)
- [ ] 모바일 "포즈 설정" 탭 레이아웃 최적화 (핵심 기능 상단 배치)
- [ ] 태그/카테고리 기반 검색 모바일 전용 진입점 추가
- [ ] 포즈 프리셋 카드 터치 타겟 44px 보장
- [ ] 이미지 상세 모달 모바일 레이아웃 개선
- [ ] 랜딩 페이지 모바일 CTA 개선 (태그 검색 페이지 직행 옵션)
- [ ] 모바일에서 3D 마네킹 대신 2D 마네킹 기본 활성화 (선택적)
- [ ] 이미지 상세 모달 스와이프 네비게이션 (터치 제스처)

- 3D 마네킹 자체의 모바일 최적화 (WebGL 성능 튜닝) — 별도 스프린트

- 새로운 이미지 DB 추가 — 별도 콘텐츠 전략
- 소셜 로그인 등 인증 개선 — 별도 스프린트
- 드로잉(sketch) 모드 개선 — 별도 스프린트

## 4. User Journey Analysis

### 4.1 현재 모바일 플로우 (As-Is)

```
앱 접속 (/mannequin)
    ↓
Three.js 로딩 (2~5초, 모바일 GPU 부하)
    ↓
상단 헤더 과밀 (로고+탭+검색+링크+로그인 → 찌그러짐)
    ↓
"포즈 설정" 탭 기본 표시
    ↓
[스크롤 상단] 3D 마네킹 뷰어 (조작 어려움, 관절 드래그 미스터치)
    ↓
[스크롤 중간] 도구 바 (작은 버튼들)
    ↓
[스크롤 하단] 포즈 프리셋 카드 (★ 실제로 쓸 기능)
    ↓
"결과" 탭 탭하기 (탭 바 위치: 상단에 있음)
    ↓
이미지 그리드 (2열)
    ↓
이미지 탭 → 상세 모달
    ↓
[스크롤 필요] 컬렉션 저장 버튼
```

**마찰 지수**: 높음 (핵심 기능까지 3단계 이상 스크롤)

### 4.2 목표 모바일 플로우 (To-Be)

```
앱 접속 (/mannequin)
    ↓
2D 마네킹 기본 로딩 (빠름) + 3D 전환 버튼 제공
    ↓
[즉시 보임] 포즈 프리셋 카드 (접이식 마네킹 뷰어 위)
    ↓
포즈 선택 (한 탭)
    ↓
"결과 (N건)" 탭 탭하기 (결과 수 즉시 보임)
    ↓
이미지 그리드 (2열, 큰 터치 타겟)
    ↓
이미지 탭 → 전체화면 모달
    ↓
[즉시 보임] 컬렉션 저장 버튼 (하단 고정)
```

**마찰 지수**: 낮음 (핵심 기능이 첫 화면에)

## 6. Risks and Mitigation

| 리스크 | 영향 | 가능성 | 대응 방안 |
|--------|------|--------|-----------|
| 모바일 레이아웃 변경이 데스크톱에 영향 | 높음 | 낮음 | `md:hidden` / `hidden md:block` 분기 철저히 유지 |
| 3D→2D 기본값 변경 시 3D 기능 인지도 하락 | 중간 | 중간 | "3D로 전환" CTA를 눈에 잘 띄게 배치 |
| 이미지 상세 모달 스와이프 구현 시 기존 스크롤과 충돌 | 중간 | 높음 | react-swipeable 등 검증된 라이브러리 사용 또는 단순 버튼 방식 유지 |
| 헤더 재설계 시 모바일/데스크톱 분기 코드 복잡도 증가 | 낮음 | 중간 | MobileHeader / DesktopHeader 컴포넌트 분리 |

## 7. Architecture Considerations

### 7.1 Project Level

**Dynamic** (기존 유지) — bkend.ai + Next.js App Router + Zustand

### 7.2 Key Architectural Decisions

| 결정 사항 | 옵션 | 선택 | 이유 |
|-----------|------|------|------|
| 모바일 헤더 분기 | 조건부 렌더링 vs 별도 컴포넌트 | 별도 MobileHeader 컴포넌트 | 가독성, 유지보수성 |
| 포즈 설정 탭 순서 | CSS order 변경 vs JSX 순서 변경 | JSX 순서 변경 + `hidden md:block` | 접근성 및 스크린리더 순서 일관성 |
| 이미지 상세 모달 | 현행 유지 + 모바일 최적화 vs 별도 모달 | 현행 `ImageGrid` 내 모바일 분기 추가 | 파일 소유권 단일 유지 |
| 스와이프 제스처 | 직접 구현(touch events) vs 라이브러리 | 직접 구현(단순 X 방향 감지) | 의존성 최소화 |
| 모바일 태그 검색 진입점 | 별도 탭 vs 하단 시트(Bottom Sheet) | 탭 바 + 필터 시트(초기 단순 구현) | 빠른 구현, 추후 확장 가능 |
